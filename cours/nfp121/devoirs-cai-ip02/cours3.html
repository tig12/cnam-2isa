Salaires
Pour pratiquer l'héritage, les classes abstraites, les interfaces.
Un énoncé copié sur internet et pas modifié se trouve sur salaires-raw.txt.
Vous pouvez adapter l'énoncé (par ex coder dans plusieurs fichiers ou faire un package).
Essayez d'écrire du code propre : bien indenté, cohérence dans les sauts de ligne, pas de vieux code inutile qui traîne dans des commentaires ; choix des noms de classes, interfaces, méthodes, variables. Imaginez que vous relisez votre code dans un an ; le code est-il lisible, peut-on comprendre facilement ce qu'il fait et comment il le fait ?
  1. Faites l'exercice en faisant un diagramme de classe (sur papier ou logiciel - par ex draw.io).
  2. Regardez si les méthodes par défaut et statiques introduites en java 8 dans les interfaces permettent de modifier votre code.
On a vu comment sont gérés les conflits de noms avec import classique (ex de java.util.List et java.awt.List).
Ecrivez du code pour observer comment import static gère les conflits de noms.
Regardez la liste des patterns, et trouvez un ou plusieurs exemples de patterns que vous avez déjà utilisé en programmation. Décrivez rapidement la problématique et la manière dont le pattern a résolu le problème.
Vous pouvez aussi proposer des patterns qui ne sont pas dans la liste historique GOF.
NB : "je ne connais pas d'exemple" est aussi une bonne réponse.
